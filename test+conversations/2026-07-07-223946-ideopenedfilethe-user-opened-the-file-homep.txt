  The core problem

  Right now every variable lives inside init_params() as a local variable:

  void init_params(int ){
      double ez[SIZE], hy[SIZE], ...;   // these are LOCAL
      int maxTime = 1000, mm;
      ...
  }   // <-- the moment this brace closes, ALL of them are destroyed

  Then main() tries to use ez, denom, sigma, dt, maxTime, mm… none of which
  exist in main's scope. So this can't work as written — the initialization
  happens in one function and the data evaporates before run_simulation() (or
  main) ever sees it.

  ★ Insight ─────────────────────────────────────
  - Local variables have automatic storage duration — they live on the stack
  frame of the function that declares them and are gone the instant that
  function returns. A pointer to them would dangle.
  - You cannot return a C array from a function (return ez; is illegal/decays to
  a dangling pointer). This is exactly why C programmers reach for a struct.
  - Passing a struct (or pointer to one) is how C keeps related state alive and
  moves it between functions cleanly.
  ─────────────────────────────────────────────────

  The idiomatic C fix: bundle state into a struct

  Instead of ~15 loose variables, you group them into one type. You create one
  struct, hand its address to init_params() to fill in, then hand the same
  address to run_simulation(). Both functions operate on the same memory.

  The skeleton looks like this:

  typedef struct {
      double ez[SIZE], hy[SIZE], jp[SIZE], old_ez[SIZE];
      double cje[SIZE], cjj[SIZE], ca[SIZE], cb[SIZE], denom[SIZE];
      double saved[SIZE];

      int    maxTime;
      double imp0, e_d, sigma, Sc, dt, abc_coef, eps0, eps_inf;
      int    Nt;
  } Simulation;

  void init_params(Simulation *sim);      // fills sim in
  void run_simulation(Simulation *sim);   // reads/updates sim

  int main(void) {
      Simulation sim;          // <-- allocated here, lives for all of main
      init_params(&sim);       // pass its ADDRESS
      run_simulation(&sim);    // same object, still alive
      return 0;
  }

  Inside the functions you reach members through the pointer: sim->ez[i],
  sim->sigma, etc. Because both functions receive the same address, whatever
  init_params writes, run_simulation reads.

  ★ Insight ─────────────────────────────────────
  - We pass Simulation * (a pointer), not Simulation by value. A by-value copy
  would duplicate ~11 arrays of 200 doubles each time — wasteful, and changes
  wouldn't propagate back to main. The pointer is 8 bytes and gives shared,
  in-place access.
  - sim->ez[i] is just shorthand for (*sim).ez[i] — "follow the pointer, then
  index the array member."
  - Constants that never change per-cell (imp0, eps0, dt…) could alternatively
  stay as #defines or const globals; putting them in the struct keeps everything
  in one place. That's a real design choice, not a rule.

  What changed and why it answers your question

  Your original code had the variables declared inside init_params, so they died
  when that function returned and main couldn't see them. The fix is the
  pattern you asked about:

  int main(void) {
      Simulation sim;        // 1. CREATE the state object (lives in main's 
  frame)
      init_params(&sim);     // 2. INITIALIZE it — pass the ADDRESS
      run_simulation(&sim);  // 3. PASS the same object to the next function
  }

  ★ Insight ─────────────────────────────────────
  - &sim passes the address, not a copy. Both init_params and run_simulation
  receive a pointer to the same Simulation, so writes in one are visible in the
  other. Passing sim by value would copy ~2400 doubles each call and the changes
  wouldn't propagate back.
  - sim is declared in main, so it outlives both calls. That's the key contrast
  with your original — the data now lives in the caller's frame, and the helper
  functions just borrow a pointer to it.
  - sim->ez[i] = (*sim).ez[i] — dereference the pointer, then index the array
  member. That arrow operator is the whole vocabulary you need for
  struct-via-pointer code.

● Now I understand your setup — and there's an important physics subtlety I need
  to flag before coding, because it determines whether your two curves will
  actually lie on top of each other.

  Here's the situation. Reading Debye.py, your intended model is the single‑pole
  Debye relaxation:

  $$\varepsilon_r(\omega) = \varepsilon_\infty + \frac{\varepsilon_s -
  \varepsilon_\infty}{1 + j\omega\tau}, \qquad \tau = N_t,\Delta t = 50$$

  and the analytic Fresnel reflection at the free‑space→Debye interface (normal
  incidence, non‑magnetic so $n=\sqrt{\varepsilon_r}$):

  $$\Gamma_{\text{analytic}}(\omega) = \frac{n_{\text{debye}} -
  n_{\text{free}}}{n_{\text{debye}} + n_{\text{free}}} =
  \frac{\sqrt{\varepsilon_r(\omega)} - 1}{\sqrt{\varepsilon_r(\omega)} + 1}$$

  To extract $\Gamma$ from the simulation, the standard trick uses exactly the
  two runs we already have — a reference subtraction:

  - free_sim (no material) records the incident pulse at a probe →
  $E_{\text{inc}}(t)$
  - disp_sim (with material) records incident + reflected at the same probe →
  $E_{\text{tot}}(t)$
  - $E_{\text{refl}} = E_{\text{tot}} - E_{\text{inc}}$, then
  $\Gamma_{\text{sim}}(\omega) =
  \mathrm{FFT}(E_{\text{refl}}),/,\mathrm{FFT}(E_{\text{inc}})$
